# Security Policy

## Responsible Deployment

HoneyTrap AI intentionally exposes services to attackers. **Run it safely**:

1. **Network isolation** — deploy on a dedicated VM, network segment, or cloud instance that cannot reach production systems.
2. **No real credentials or data** — all content is synthetic and generated by the framework.
3. **Outbound firewall rules** — block outbound traffic from the honeypot host (except GeoIP/LLM API endpoints if used).
4. **Rate limiting** — the framework applies per-IP caps, but consider kernel-level limits too.
5. **Resource caps** — set container/VM memory and CPU limits; the honeypot is designed to never saturate a host, but defense in depth applies.
6. **Monitoring** — watch for your honeypot being used as a pivot; unusual outbound patterns are a red flag.
7. **Do not deploy on corporate production networks.** This includes home networks connected to sensitive devices.

## Reporting a Vulnerability

If you discover a security issue in HoneyTrap AI itself (for example, a sandbox escape, an unintended remote code execution path in a handler, or a credential leak), please report it privately:

- Email: **security@redhoundinfosec.io**
- Or open a GitHub Security Advisory at <https://github.com/redhoundinfosec/honeytrap-ai/security/advisories/new>

Please include:

- Affected version
- Reproduction steps
- Impact assessment
- Suggested fix (if known)

We aim to acknowledge within 72 hours and ship a patch within 14 days for high-severity issues.

## Supported Versions

Only the `main` branch and the latest tagged release receive security updates during beta.
